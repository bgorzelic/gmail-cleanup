## Key abstractions

### Three-bucket world model

Every sender falls into one of three buckets, determined by the YAML lists:

1. **Real humans** (`humans.yaml`) → starred, marked important, spam-shielded. Never auto-archived.
2. **Must-keep automation** (`keep.yaml`) → bank/health/gov/security. Substring match. The unsubscribe command *physically refuses* to act on these.
3. **Noise** (`kill.yaml` or auto-discovered) → one-click unsubscribe + archive.

The order matters: humans win over kill, kill wins over auto-discovery, keep blocks unsubscribe entirely.

### The autopilot pipeline

`gmail-cleanup autopilot` is a 4-phase composition:

1. **Apply filters** (`filters apply`) — declares categorization rules to Gmail (idempotent)
2. **Unsubscribe** (`unsubscribe --days 30 --min-count 2`) — recent noise senders
3. **Mark-read** (`mark-read --query "is:unread -in:inbox"`) — clear backlog
4. **Verify** (`verify --days 14`) — check stickiness; `--escalate` to block

Each phase can also run standalone. Autopilot is just the convenience wrapper.

### State machine for a sender

```
   appears in inbox
        │
        ▼
   has List-Unsubscribe header?
   │              │
   no             yes
   │              │
   ▼              ▼
   skip          in keep.yaml?
                 │         │
                 yes       no
                 │         │
                 ▼         ▼
                 refuse    in kill.yaml OR sender_count >= min_count?
                 (safe)    │              │
                           yes            no
                           │              │
                           ▼              ▼
                           one-click      skip (don't archive an unknown sender)
                           unsubscribe
                           + archive
                           + add to unsubbed.yaml
```

## Adding a new command — the recipe

1. Pick a name. Add `cmd_X(args)` function to `gmail_cleanup/__init__.py` near similar commands.
2. Add a parser block in `main()` after similar parsers. Set `func=cmd_X`.
3. If the command doesn't need email auth (e.g., a config helper), add it to the email-exemption tuple in `main()`.
4. If you're adding a long-running loop, use `progress_for()` / `advance()` from `gmail_cleanup.progress`.
5. Write tests. Mock `GmailCLI.search_messages` etc. — never hit real Gmail in tests.
6. Update README's command table and CHANGELOG.

## Safety invariants — DO NOT BREAK

These are enforced by `tests/test_safety.py`. Touching the substring-match in `cmd_unsubscribe` requires updating those tests with the new semantics.

1. **`UNSUB_KEEP_LIST` is honored before any unsubscribe action.** Any sender matching it (substring) is refused.
2. **Senders without `List-Unsubscribe` are skipped, not archived.** Guard against archiving humans whose mail happens to lack a header.
3. **`humans.yaml` entries get filter actions that ADD `STARRED`+`IMPORTANT`** — they should never be in the same filter that adds INBOX-removal labels.
4. **Block filters (auto-trash)** can only be created via `verify --escalate`, never silently.
